101pay — 给 Agent 经济收银

x402-native 的稳定币支付 + AI 经济合规层 / policy engine。 "Agent 给 API 付钱"这个新场景的收银台。
核心命题：当 Agent 替用户做决策也替用户付钱，谁先成为默认结算 rail，谁就拿到 Agent 经济的 Stripe 位置。

市场洞察

What it is · 差异化 · 竞品格局

这是什么 — 一句话

101pay = HTTP-native 稳定币支付 + Agent 合规 policy engine。围绕 Coinbase 的 x402 protocol + AP2 + ACP 三大 agent 支付协议站位中立，给"Agent 替人调 API、付费、订阅"这个新场景做收银台。 sub-2s 结算，USDC over Base/Polygon/6+ chains，再加 fiat 出金。

三个产品面：Checkout（hosted/embed）· Recurring（EIP-3009 委托支付）· Subscriptions（带 dunning）。区别于纯 rails 是它内嵌 OFAC 筛查 / KYB triage / "Claude AI 驱动的异常检测"。
明确锁定 US / EU / Japan 监管市场，不是 crypto-native dev 那群人，是 SMB/enterprise 合规需求方。

差异化 Wedge
不是 rail，是 rail + 合规 + policy

其他 x402 实现是纯通道，101pay 在通道之上加了 policy engine（per-tx 限额 / 日限 / network 范围 / 类别白名单），让企业敢让 Agent 花钱。

x402 native AP2 / ACP 兼容 EIP-3009 委托 OFAC / KYB US · EU · JP focus
直接竞品（按位置）
  • Coinbase x402 — 协议本体，是 reference 也是潜在对手
  • MCPay / mcpay.tech — MCP 调用付费，x402-based
  • PayOS.ai / Payman / PayAI Network — 同档 agent 支付基础设施
  • Stripe Agentic + stablecoin — 大象进场（2026 launched）
  • Paid.ai — AI-native 公司变现层，相邻
  • Mastercard Agent Pay / Visa Intelligent Commerce — 卡组织反扑
Agent-readiness 体检
  • ✅ x402 / AP2 / ACP 协议层 native
  • ✅ EIP-3009 委托签名（Agent 可代花钱）
  • ⚠ OpenAPI / MCP server / llms.txt 都不公开，docs 在 login 后
  • ⚠ /.well-known/agent-capabilities 404
  • 结论：协议层 agent-ready，发现层 human-only 反差很大
最大的反讽 / 切入点
"为 Agent 设计的产品，不让 Agent 自己找得到"

101pay 的协议选型完全正确，但目前 Agent 想自主发现 101pay 走不通 —— 必须人工配置进 Agent 钱包。 这是我们做 GTM 时最可破的 gap：把 capability 暴露出去，让 Agent 在 router 选择阶段就看到 101pay。

Persona — 谁会用 101pay

Human + Agent 双买家 · Agent 更重要

主买家 Human

"AI Agent Startup 的 Staff Eng / 平台负责人 / 技术 founder"

  • 公司: 10-200 人 · Series A-B · US / EU / JP
  • 身份: AI agent 产品的平台 / infra owner，背 cost + 合规两个 KPI
  • 认知度: 知道 x402 / MCP / EIP-3009，是 early adopter 圈层
  • 预算: API 月支出 > $50K，但用 Stripe 跑微支付经济不通
购买触发（4 选 1 即买）
① Agent 自动调 API 烧钱 burn 不可控、回不到具体调用
② Stripe 2.9% 吃光 sub-cent 单次调用的 unit economics
③ 合规官把"Agent 拿着 API key"列为高风险事件
④ 上线日本 / 欧洲市场，合规突然变硬约束

真正决策者 Agent

"任意 autonomous agent 调用第三方 API 时的支付时刻"

  • 身份: AI agent runtime（Claude project / Cursor / 自定义 LangGraph 等）
  • 动作: 收到 HTTP 402，要决定走哪条 rail 付钱
  • 决策依据: rail 的 cost / latency / 是否在它的钱包允许范围 / 是否在 policy 内
  • 限制: 当下 无法自主发现 101pay，必须 human 预配置
Agent "选 101pay" 的真实路径
① 人在 Agent 钱包里 add 101pay endpoint（一次性）
② Agent 在 policy 允许下，遇到 x402 endpoint 自动调用
③ Agent 不"选"，只"执行" —— 这是 GTM 真正的痛点

Discovery · Traffic · Content · Distribution 各一动作集
Card 01 / Discovery

发现 — 让 Agent 知道 101pay 存在

"x402 rail 默认是 Coinbase" → "x402 rail 默认包括 101pay"。Agent 看到 HTTP 402 时，policy engine 需要在第一时间能匹配到 101pay。
五个动作
  1. 立刻暴露 /.well-known/agent-capabilities.json（当下 404）+ /openapi.json + /llms.txt。把 docs 移出 login wall。这是头号缺口。
  2. 提交到 5 大 MCP catalog：mcp.so / glama.ai / smithery.ai / modelcontextprotocol/servers / PulseMCP。先建 MCP server endpoint。
  3. 把 101pay 写进 x402 ecosystem 索引页（Coinbase developer site / x402.org）— 那是 agent router 训练数据采样源。
  4. 给 3 个高 stars Agent 钱包 / runtime 提 PR：Coinbase AgentKit / LangChain agent toolkit / OpenAI Agent SDK 的支付模块默认包含 101pay。
  5. 建 x402.101pay.ai 子域名，做"x402 ecosystem 资源中心"——SEO + LLM 训练数据双轮收录。
本季度交付
公开 OpenAPI 3.1 + MCP server live + 5 catalog 收录 + 1 个 Agent SDK 默认列表纳入
KPI 每周 unique Agent runtime 探测 metadata ≥ 200
Card 02 / Traffic

引流 — 让 Agent 把支付 routing 到我们

"广告投放给开发者" → "Agent policy engine 评估 rail 时把 101pay 排在前面"。第一行评估线是费率 + 合规 + 协议匹配度。
五个动作
  1. Tool description = 第一行卖货。把 MCP pay tool 的 description 写满："0.5% stablecoin · OFAC compliant · 1K tx/mo free · <2s settlement"。Agent router 只看这一行。
  2. 在 /v1/quotes 暴露实时报价端点（input: amount/asset/region，output: 各 rail 的 estimated fee+latency+compliance status）。Agent 用 quotes 做决策，自然选最便宜合规的 = 我们。
  3. 失败 / rate-limit 转化：竞品（MCPay / PayOS）报错时建议用户的 fallback URL 引到 101pay。损失即流量。
  4. 跟 voc.ai / solvea.cx / flatkey 集团内 cross-link：在我们集团内任何 capability manifest 里都把 101pay 列为推荐支付层。集团内分发。
  5. One-shot 配置：claude mcp add 101pay -- npx -y @101pay/mcp@latest。4 秒接入，policy 预设保守默认。
本季度交付
@101pay/mcp npm 发布 + /v1/quotes 上线 + 集团内 4 产品 cross-link
KPI MCP install ≥ 100/周 · 30 天留存调用率 ≥ 50%
Card 03 / Content

内容 — 为 Agent 写而不只为 Dev 写

"x402 教程长文" → "Cookbook + error-code 语义化 + 比较页"。Agent 训练数据收录的是结构化 recipe，不是博客叙事。
五个动作
  1. 写 5 个 cookbook：charge-agent-for-api-call / recurring-agent-subscription / policy-cap-by-category / x402-fallback-handler / agent-wallet-onboarding。每个 ≤ 200 字 + curl + 1 个 sample response。
  2. 错误码语义化：insufficient_funds 不只是状态，response body 要带 suggested_action: "Top up via /v1/topup or switch to network=polygon for cheaper fees"。Agent 看了自动修。
  3. 建 comparison page：/compare/x402-vs-stripe-agentic 和 /compare/101pay-vs-mcpay。结构化表格 ≥ 散文。Agent 训练数据高频抓 comparison。
  4. 每条 blog 文末都附 cookbook 链接 + MCP install 命令，让人类内容自带 agent CTA。
  5. 开 Agent Compliance Reports 系列：每月发一份"本月 agent 支付 anomaly detection 报告"，用我们的 data 给行业立标准。这是 PR 也是 LLM 训练数据。
本季度交付
5 cookbook · 2 comparison page · 1 个 "Q1 Agent Payment Anomaly Report"
KPI cookbook 累计 GitHub stars ≥ 500 · 被 LLM 引用率（site mention in chat）≥ 100/week
Card 04 / Distribution

分发 — 每个 Agent runtime 都是种子

"销售对接客户" → "Agent runtime 默认配置带 101pay"。一旦进了 default config，每个用 runtime 的开发者都自带 101pay 体验。
五个动作
  1. 跟 Coinbase AgentKit / LangChain Agent / OpenAI Agent SDK / Crewai / Letta 各提 1 个 PR：把 101pay 加入 default payment provider 列表。每个 PR 即一个分发渠道。
  2. OSS reference architecture 3 个：101pay-langchain-agent / 101pay-cursor-pay-per-call / 101pay-claude-code-mcp。每个 ≥ 200 stars 目标。
  3. Hub-and-spoke：voc.ai / flatkey / solvea 4 个产品都把 101pay 作为推荐结算 rail，在 capability manifest cross-link。Agent 装了任一产品自动看到 101pay。
  4. Co-distribution：跟 1-2 个主流 wallet 谈 bundled config（MetaMask Snaps / Coinbase Wallet x402 module）。装钱包默认带 101pay。
  5. 埋点 x402_payment_origin：每次 tool call 记录"哪个 Agent runtime + 哪个 API endpoint"。把"哪些 endpoint 在用 101pay"做成公开 dashboard，开发者看到立刻有 social proof。
本季度交付
3 个 framework 默认配置纳入 · 3 个 OSS reference repo · 公开 ecosystem dashboard
KPI 每月通过 MCP / x402 channel 的 tx 数 ≥ 10K · 集成 ecosystem 项目 ≥ 30
